<objective>
Create the `app-v*`-tag-triggered GitHub Actions workflow that builds both Zig helper bundles, signs+notarizes (when secrets present), repackages them as `Shoki.app` + `Shoki Setup.app` inside a unified `shoki-darwin-<arch>.zip`, computes a SHA256 sidecar, and publishes both files to a GitHub Release. Plan 02's `shoki setup` is the consumer.

Purpose: This workflow is the producer side of the download flow. The SDK + CLI (Plan 02) and the slimmed binding tarballs (Plan 04) only make sense if the matching release artifacts exist on GitHub. This plan ships them.

Output: A green `app-release.yml` workflow + a documented release procedure. After this plan, tagging `app-v0.1.0` produces a release containing exactly `shoki-darwin-arm64.zip`, `shoki-darwin-arm64.zip.sha256`, `shoki-darwin-x64.zip`, `shoki-darwin-x64.zip.sha256`.
</objective>

  <action>
    Write `helper/scripts/package-app-zip.sh` with `set -euo pipefail` and the following body:

    1. Parse `--arch <arm64|x64>` (required) and `--out-dir <path>` (optional, default `helper/.build`).
    2. Resolve `HELPER_DIR` as `$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)`.
    3. Verify `$HELPER_DIR/.build/ShokiRunner.app` and `$HELPER_DIR/.build/ShokiSetup.app` exist; fail loudly if not.
    4. Stage to a temp dir (`mktemp -d`) — do NOT mutate the build outputs in place:
       - `ditto "$HELPER_DIR/.build/ShokiRunner.app" "$STAGE/Shoki.app"` (rename)
       - `ditto "$HELPER_DIR/.build/ShokiSetup.app" "$STAGE/Shoki Setup.app"` (rename + space)
    5. Package: `(cd "$STAGE" && ditto -c -k --keepParent "Shoki.app" "Shoki Setup.app" "$OUT_DIR/shoki-darwin-$ARCH.zip")`
       - Note: `ditto -c -k --keepParent` with multiple sources packs both .app trees with their top-level dir intact at the zip root.
    6. Compute SHA256 with `shasum -a 256 "$OUT_DIR/shoki-darwin-$ARCH.zip" | awk '{print $1 "  shoki-darwin-'"$ARCH"'.zip"}' > "$OUT_DIR/shoki-darwin-$ARCH.zip.sha256"`
       - The two-space separator matters — that's the format Plan 02's parser handles.
    7. Print the two output paths and the sha256 hex to stdout for log visibility.

    Make the script executable: `chmod +x helper/scripts/package-app-zip.sh`.

    Local sanity test (run on a darwin host): `cd helper && ./scripts/build-app-bundle.sh && ./scripts/package-app-zip.sh --arch arm64`. The two output files must appear in `helper/.build/`.
  </action>

    Add `.github/workflows/app-release.yml` with three jobs:

    Trigger:
    ```yaml
    on:
      push:
        tags: ['app-v*']
      workflow_dispatch:
        inputs:
          version:
            description: 'Version (e.g. 0.1.0) — only used by manual dispatch'
            required: true
    permissions:
      contents: write   # for gh release create
    concurrency:
      group: app-release-${{ github.ref }}
      cancel-in-progress: false
    ```

    Job: `build-arm64` (runs on macos-14):
    1. checkout
    2. mlugg/setup-zig@v2 with version 0.16.0
    3. (Conditional) Import Developer ID cert via apple-actions/import-codesign-certs@v3 ONLY if `secrets.APPLE_DEVELOPER_ID_APPLICATION_CERT_B64` is set; otherwise skip and ad-hoc sign downstream
    4. `cd helper && zig build -Doptimize=ReleaseFast -Dtarget=aarch64-macos`
    5. `cd helper && ./scripts/build-app-bundle.sh` (verifies bundles)
    6. (Conditional) Sign + notarize both bundles via `helper/scripts/sign-and-notarize.sh` ONLY if `secrets.DEVELOPER_ID_IDENTITY` is set. Use the same env-var contract as `release.yml` (DEVELOPER_ID_IDENTITY, APPLE_TEAM_ID, APPLE_API_KEY_ID, APPLE_API_ISSUER_ID, APPLE_API_KEY_B64).
    7. `cd helper && ./scripts/package-app-zip.sh --arch arm64` → produces `helper/.build/shoki-darwin-arm64.zip` + `.sha256`
    8. actions/upload-artifact@v4 with `name: app-darwin-arm64`, paths to both files

    Job: `build-x64` (runs on macos-14, cross-compile from arm64 host):
    - Same as build-arm64 but `-Dtarget=x86_64-macos` and `--arch x64`. (The Zig helper builds both targets cleanly from macos-14; release.yml proves this.)

    Job: `publish-release` (runs on ubuntu-latest, needs: [build-arm64, build-x64]):
    1. checkout
    2. actions/download-artifact@v4 for both `app-darwin-arm64` and `app-darwin-x64` (download into `dist/`)
    3. Resolve VERSION from `${{ github.ref_name }}` (strip `app-v` prefix) or from `${{ inputs.version }}` for manual dispatch
    4. `gh release create "app-v$VERSION" dist/app-darwin-arm64/shoki-darwin-arm64.zip dist/app-darwin-arm64/shoki-darwin-arm64.zip.sha256 dist/app-darwin-x64/shoki-darwin-x64.zip dist/app-darwin-x64/shoki-darwin-x64.zip.sha256 --title "Shoki.app v$VERSION" --notes "Native helper bundles for shoki SDK. Install via \`npx shoki setup\` — manual download not required."`
    5. `env: GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}`
    6. Use `--target ${{ github.sha }}` so the release points at a real commit on the tag

    Critical detail: this workflow is INDEPENDENT of `.github/workflows/release.yml`. The `v*` tag publishes the SDK + bindings to npm; the `app-v*` tag publishes the helper bundles to GitHub Releases. Two separate cadences — the SDK can release patches without re-cutting the helper bundle, and vice versa.

    Document the new workflow in `docs/background/release-setup.md`. Add a new section `## App release (helper bundles)` after the existing SDK section explaining:
    - When to bump (only when `helper/` source changes)
    - How to tag (`git tag app-v0.1.0 && git push origin app-v0.1.0`)
    - The relationship to `compatibleAppVersion` in `packages/sdk/package.json` — they MUST match. SDK release cadence depends on helper release cadence; bumping the helper requires bumping the SDK's `compatibleAppVersion` and re-publishing the SDK so consumers fetch the right release.
    - The artifact contract (zip + sha256, two architectures)
