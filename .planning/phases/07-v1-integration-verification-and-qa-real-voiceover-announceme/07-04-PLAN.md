<objective>
Prove — and build the mechanism to prove — that what shoki captures is
actually announcements from DOM content the test exercises, not incidental
chrome noise (URL bar, tab title, address-bar autofill). CONTEXT.md calls
this "the most important functional requirement" of Phase 7.

The approach, per CONTEXT.md's analysis, combines:
1. **Filter by renderer pid** — AX observer subscribes only to the Chromium
   renderer process that owns the test viewport, not the parent Chromium
   chrome process.
2. **Focus the content area** before capture starts — so VO cursor lands in
   the page, not the window chrome.
3. **Document the pattern** in matchers.md so users writing tests know
   the expected workflow.
4. **Verify with paired tests**: a magic string in the URL-bar-only-path
   must NOT appear in the log; the same string in the DOM must appear.

Output: Helper + Zig side change to scope AX observation to the renderer
pid; example test pair proving the filter works; docs update.
</objective>

From zig/src/drivers/voiceover/driver.zig lines 122-129 + 196-207:
- startImpl currently resolves the VoiceOver process pid via
  `resolveVoiceOverPid` and hands THAT pid to AxNotifications.
- AX observer is configured with the VO-process pid; that observer sees
  EVERY announcement VO makes from any source.
- **The problem:** it does NOT currently filter by which app is being
  observed. We can't simply "filter AX notifications by pid" on the
  VO-process side — VO itself is a single process that announces things
  from many source apps. We need a different filter.

**Revised filter strategy:** AXObserver can take an element as its target
(AXUIElementCreateApplication(pid)). Instead of observing `system-wide`
(the current Phase 3 approach), observe only the Chromium renderer
process's AXUIElement. That way, only AX events from THAT app reach the
callback.

From helper/Sources/ShokiRunnerService/AXObserverSession.swift (not
re-read here — Plan 04 in Phase 3 landed this):
- Current implementation calls `AXUIElementCreateSystemWide()` and attaches
  AXObserverCreateWithInfoCallback to it. System-wide == every app.
- To scope, change to `AXUIElementCreateApplication(renderer_pid)`.

From examples/vitest-browser-react/src/App.tsx:
```tsx
export function App() {
  const [message, setMessage] = useState('');
  const onSubmit = () => { setMessage(''); requestAnimationFrame(() => setMessage('Form submitted')); };
  return ( <main><h1>Shoki Vitest Browser Example</h1>
      <form onSubmit={e => {e.preventDefault(); onSubmit();}}>
        <button type="submit">Submit</button>
      </form>
      <p aria-live="polite" role="status">{message}</p>
  </main>);
}
```
We extend this with a simple path-based router (no react-router dep —
read `window.location.pathname` directly) so `/` renders the default and
`/test-page-xxyyzz-DOM` renders the magic-marker text.

Resolving the renderer pid is the trickiest part. Options:
(A) Playwright exposes `page.context().browser().newBrowserCDPSession()` →
    `Target.getTargetInfo` includes the browser pid. Chromium's renderer
    pid is a child of that.
(B) `ps -o pid,ppid,comm | grep -E 'Chromium Helper \\(Renderer\\)'` —
    heuristic but works on macOS.
(C) NSRunningApplication + filter by bundle-id + process name — matches
    the parent Chromium, not the renderer.

Option (A) is the most reliable but requires a Playwright dep in the
browser-side bridge. Option (B) is Zig-side and avoids any Playwright
import at the SDK layer. This plan uses (B) — a small Zig helper that
runs `pgrep -f "Chromium Helper.*Renderer"` and returns the child pids.
If multiple renderer children exist (mutiple tabs), we take the one whose
parent matches the specific test's browser pid (Plan 07-06 wires the
browser pid through if needed; for this plan, we accept that the single
test-mode browser run has exactly one renderer).

**Fallback if (B) doesn't uniquely resolve:** The test focuses the page
viewport via `await page.locator('body').focus()` before capture; this
alone keeps VO cursor in the page in practice. Option (A) pid-filter is
defense-in-depth. The two tests verify this combined approach works:
the negative test fails if EITHER mechanism alone is insufficient.

  ```zig
  pub fn resolveChromeRendererPid(allocator: std.mem.Allocator, runner: defaults_mod.SubprocessRunner) !i32 {
      const argv = [_][]const u8{ "/usr/bin/pgrep", "-f", "Chromium Helper (Renderer)" };
      var result = try runner.run(allocator, &argv);
      defer result.deinit(allocator);
      if (result.exit_code != 0) return error.RendererNotFound;
      const trimmed = std.mem.trim(u8, result.stdout, " \t\r\n");
      // Take the LAST line (youngest pid = most recently spawned, usually the Vitest-spawned tab).
      var last: ?[]const u8 = null;
      var it = std.mem.splitScalar(u8, trimmed, '\n');
      while (it.next()) |line| last = line;
      const s = last orelse return error.RendererNotFound;
      return std.fmt.parseInt(i32, s, 10);
  }
  ```

  ```markdown
  ## Chrome noise: how to avoid capturing URL-bar text

  When Vitest browser-mode opens Chromium via Playwright, VoiceOver sees
  the entire OS screen — including the URL bar, tab titles, and address-bar
  autofill. If your assertion naively checks that a specific string was
  announced, you may get a false positive from chrome text.

  Shoki's recommended pattern (the canonical example's
  `dom-vs-chrome-url.test.tsx` demonstrates it):

  ### 1. Scope AX capture to the renderer process

  Shoki honors `SHOKI_AX_TARGET_PID` as the target pid its AX observer
  binds to. Set it before `voiceOver.start()` runs:

      const out = execFileSync('/usr/bin/pgrep', ['-f', 'Chromium Helper (Renderer)'], { encoding: 'utf8' });
      process.env.SHOKI_AX_TARGET_PID = out.trim().split('\n').pop()!;

  When set, AX notifications from the parent Chromium process (which
  includes the URL bar, tab bar, and window chrome) are excluded from
  capture.

  ### 2. Focus the page viewport before capture

  VO's AppleScript "last phrase" capture path doesn't respect process
  boundaries the same way AX notifications do. It follows VO cursor.
  Before your assertion, focus the page content so VO cursor lands there:

      await page.locator('body').focus();
      await session.reset();
      // Now perform your action and assert.

  ### 3. Use `toHaveAnnounced({ role, name })` instead of string-matching

  Structured matchers are immune to most chrome noise because the parent
  Chromium process's elements don't have the same (role, name) shape as
  a `<button>Submit</button>` in the DOM:
  ```
