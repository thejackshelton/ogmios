    # Shoki

    > **Status: Phase 1 — Foundations. Not usable yet. First npm release lands at the end of Phase 3.**

    Shoki lets you run **real** screen readers — VoiceOver, and more to come — inside any CI environment and any test framework. It's not a test runner and not a static checker: you start a screen reader from your existing tests, capture everything it would have announced, and assert on that. Think of it as a more ambitious [Guidepup](https://github.com/guidepup/guidepup) with first-class CI support and dramatically better DX.

    **Core value:** a test author can start a real screen reader from their existing test framework, capture what it announces, and assert on it — locally and in CI — without becoming a sysadmin.

    ## What Phase 1 delivers

    Phase 1 is the trust-anchor foundation. When it's done:

    - `@shoki/sdk` is installable from npm; it auto-loads a signed, platform-specific native addon via `optionalDependencies` (`@shoki/binding-darwin-arm64` or `@shoki/binding-darwin-x64`).
    - Every released binary is Apple Developer ID-signed (`codesign -dvvv` shows the cert).
    - A signed `ShokiRunner.app` bundle ships inside the binding package as the **TCC trust anchor** — TCC grants survive Node upgrades because the bundle signature stays stable (see [`docs/architecture.md`](./docs/architecture.md)).
    - A frozen `ShokiDriver` vtable (Zig) + `ScreenReaderHandle` interface (TS) means future drivers slot in with one new `src/drivers/<name>/driver.zig` file and one registry entry — no core or SDK edits.
    - Pushing `v*.*.*` triggers CI that cross-compiles Zig 0.16+ for both darwin triples and publishes via OIDC trusted publishing (no `NPM_TOKEN`).

    **What does NOT work yet:** VoiceOver capture, `shoki doctor`, Vitest integration, CI image. Those land in Phases 2–5.

    ## Phase 1 smoke test

    After a release (or a local build — see [`CONTRIBUTING.md`](./CONTRIBUTING.md)):

    ```bash
    npm install @shoki/sdk
    node -e "console.log(require('@shoki/sdk').ping())"
    # pong
    ```

    That round-trip proves: native addon loaded → N-API dispatch → Zig core → round-trip back. Everything from Phase 3 onward builds on top of this.

    - **Node:** 24 LTS (`.nvmrc` pins this)

    ```bash
    pnpm lint                 # biome check (entire workspace)
    ```

    > **Scope:** this is the short, public-facing architecture reference for Phase 1.     ## Three layers, one process

    Shoki is in-process. Test runner, SDK, and native core all live in the same Node worker. The only cross-process boundary at runtime is the helper app that holds TCC identity.

        ┌───────────────────────────────────────────────────────────────────┐
        │ Node / Bun worker (Vitest, Playwright, Jest — whatever)            │
        │                                                                    │
        │   ┌────────────────────────┐                                      │
        │   │ @shoki/sdk (TypeScript)│  voiceOver()    → ScreenReaderHandle │
        │   └───────────┬────────────┘                                      │
        │               │ N-API                                             │
        │               ▼                                                   │
        │   ┌────────────────────────┐                                      │
        │   │ shoki.node (Zig)       │  ShokiDriver vtable, ring buffer,    │
        │   │                        │  wire format, driver registry        │
        │   └────────────┬───────────┘                                      │
        │                │  NSXPCConnection                                 │
        └────────────────┼──────────────────────────────────────────────────┘
                         ▼
        ┌──────────────────────────────────────────────────────────────────┐
        │ ShokiRunner.app  (Developer ID-signed, notarized)                │
        │   NSXPCListener("org.shoki.runner")                              │
        │   Phase 1: ping()                                                │
        │   Phase 3: + startVoiceOver / stopVoiceOver / getLastPhrase ...  │
        │                                                                  │
        │   → osascript ↔ VoiceOver.app (Phase 3)                          │
        └──────────────────────────────────────────────────────────────────┘

    ## Package topology

    | Package | Role |
    |---|---|
    | `@shoki/sdk` | TypeScript public surface. Loads the platform binding via `optionalDependencies`. Exports `voiceOver()`, `ping()`, `version()`, `ScreenReaderHandle`, `ShokiEvent`. |
    | `@shoki/binding-darwin-arm64` | Apple Silicon native addon (`shoki.node`) + signed `helper/ShokiRunner.app`. |
    | `@shoki/binding-darwin-x64` | Intel native addon + signed helper. |

    On install, npm resolves only the platform's matching binding. No postinstall scripts. No runtime downloads. (Same pattern as esbuild, swc, rolldown, napi-rs v3 — see [PITFALLS.md Pitfall 12](../.planning/research/PITFALLS.md).)

    ## Driver interface (EXT-01)

    Adding a new screen reader touches three things, in this order:

    1. A new `zig/src/drivers/<name>/driver.zig` that implements the `ShokiDriver` vtable.
    2. One new entry in `zig/src/core/registry.zig`.
    3. One new `@shoki/binding-<platform>-<arch>` package + one row added to `.github/workflows/release.yml`'s matrix.

    Zero changes to `core/`, the SDK, or the wire format. This is EXT-01 by construction — see [`docs/adding-a-driver.md`](./adding-a-driver.md) for the step-by-step.

    ## Signed-wrapper-app architecture (FOUND-04 decision record)

    **Decision:** TCC is granted to a Developer ID-signed, notarized `ShokiRunner.app` bundle, NOT to Node, NOT to the user's terminal.

    **Why:**
    - TCC identifies processes by **code signature**, not by name. Node's binary hash changes between minor versions (22.1 → 22.2, etc.) and on most user machines Node is unsigned. Granting TCC to Node = a permission re-prompt on every Node upgrade.
    - Granting TCC to the user's terminal is intrusive (the terminal is now privileged for every script, not just Shoki) and doesn't help CI at all.
    - A signed helper app controlled by Shoki gives TCC a **stable identity** across Node upgrades, Zig rebuilds, and SDK patches. Users re-grant only when Shoki rotates its Developer ID cert — which we announce in release notes.

    **How:**
    - `helper/` is a Swift package producing `ShokiRunner.app` with mach service name `org.shoki.runner`.
    - CI signs the bundle with Developer ID Application, notarizes with `notarytool`, and ships the signed `.app` inside each `@shoki/binding-*` package.
    - The Zig `.node` addon spawns / connects to the helper via `NSXPCConnection`. VoiceOver AppleScript runs inside the helper's process — so TCC sees the helper's stable signature.

    **Phase 1 scope:** helper exists, is signed, is notarized, and answers `ping()` over XPC. Real VoiceOver calls route through it in Phase 3.

    Full threat model and alternatives considered in [PITFALLS.md Pitfall 4](../.planning/research/PITFALLS.md) and [research/ARCHITECTURE.md](../.planning/research/ARCHITECTURE.md).

    ## Wire format

    Binary buffer drained from Zig and decoded in TS:

        [u32 WIRE_VERSION=1][u32 count][entry]*
        entry = [u64 ts_nanos LE]
                [u8 source_tag][u8 flags]
                [u16 phrase_len LE][phrase utf8]
                [u16 role_len LE][role utf8?]
                [u16 name_len LE][name utf8?]

    Source tags: 0 = applescript, 1 = ax, 2 = caption, 3 = commander, 255 = noop.

    **Frozen at end of Phase 3.** Phase 1 writes `WIRE_VERSION = 1` and a regression test that fails on any field-width or ordering change.

    ## Process model + polling (pointers for Phase 3)

    - Capture loop (Phase 3) lives inside the Zig `.node`, not in JS. Poll cadence 50ms.
    - Ring buffer (bounded 10k entries default) in Zig; TS calls `drain()` to pull encoded entries.
    - Dropped entries increment `droppedCount` rather than OOM the worker.
    - Dual capture paths in Phase 3: AppleScript (primary) + AX notifications (CVE-2025-43530 hedge). Both feed the same ring buffer via the `source` tag.

    For the full model including Vitest browser-mode topology (WebSocket → Node → N-API → Zig → XPC → VoiceOver), see [research/ARCHITECTURE.md](../.planning/research/ARCHITECTURE.md).

    ## Not here (by design)

    - Driving the app (clicks, typing, rotor nav). User's existing test framework drives; Shoki is observe-only in v1.
    - Audio-capture / ASR of VO output. Disproportionate cost, probabilistic fidelity. Flagged post-v1.
    - Notarization of the `.node` addon. npm-distributed `.node` files inherit trust from Node. Only the helper app is notarized.
    - Out-of-tree driver loading via `dlopen`. Post-v1; revisit if a concrete request surfaces.

    # Adding a screen reader driver

    Shoki's extensibility contract (EXT-01): **adding a new screen reader must NOT require changes to the core, SDK, or wire format.**

    This walkthrough implements a hypothetical `orca` driver (Linux). Same shape applies for NVDA (Windows), iOS VoiceOver via XCUITest, etc.

    ## Prerequisites

    - Working Shoki dev environment (see [`CONTRIBUTING.md`](../CONTRIBUTING.md))
    - Read [`zig/src/drivers/README.md`](../zig/src/drivers/README.md) — that's the source-tree-level contract
    - Read [`docs/architecture.md`](./architecture.md) — understand where your driver fits

    ## Step 1 — Implement the vtable in Zig

    Create `zig/src/drivers/orca/driver.zig`:

    ```zig
    const std = @import("std");
    const driver_mod = @import("../../core/driver.zig");
    const opts_mod = @import("../../core/options.zig");

    pub const OrcaDriver = struct {
        allocator: std.mem.Allocator,
        // your state fields

        pub fn create(allocator: std.mem.Allocator) !*OrcaDriver { ... }
        pub fn vtable() driver_mod.ShokiDriver {
            return .{
                .init = initImpl,
                .start = startImpl,
                .stop = stopImpl,
                .drain = drainImpl,
                .reset = resetImpl,
                .deinit = deinitImpl,
                .name = "orca",
                .platform = .linux,
            };
        }
        // fn initImpl(...) anyerror!void { ... }
        // ...
    };
    ```

    Match the noop driver's layout at [`zig/src/drivers/noop/driver.zig`](../zig/src/drivers/noop/driver.zig). Every pushed `Entry` MUST set a valid `source` tag — reuse one of the existing values or add a new one to `zig/src/core/options.zig` **in coordination with the wire format version**.

    ## Step 2 — Register the driver

    Edit `zig/src/core/registry.zig`. Add your factory and one array entry:

    ```zig
    const orca_mod = @import("../drivers/orca/driver.zig");

    fn makeOrca(allocator: std.mem.Allocator) anyerror!driver_mod.DriverHandle {
        const instance = try orca_mod.OrcaDriver.create(allocator);
        const vt_slot = try allocator.create(driver_mod.ShokiDriver);
        vt_slot.* = orca_mod.OrcaDriver.vtable();
        return .{ .ctx = @ptrCast(instance), .vtable = vt_slot };
    }

    pub const drivers = [_]RegisteredDriver{
        .{ .name = "noop",      .platform = .any,    .create = makeNoop },
        .{ .name = "voiceover", .platform = .darwin, .create = makeVoiceOver },
    ```
