<!-- Naming convention clarification. -->

CONTEXT.md §Helper Application says the .app bundle filenames become `Munadi.app` + `Munadi Setup.app`. But the existing pattern is:
- `ShokiRunner` (executable + .app folder name, no space)
- `ShokiSetup` (executable + .app folder name, no space)
- `Shoki Runner` / `Shoki Setup` as CFBundleDisplayName (the user-visible name with space)

We keep the existing pattern and normalize on:
- Executables and `.app` bundle directory names: `MunadiRunner`, `MunadiSetup`, `MunadiRunner.app`, `MunadiSetup.app` (PascalCase, no spaces)
- CFBundleDisplayName (the label in macOS UI): `Munadi Runner`, `Munadi Setup` (with spaces)
- There are TWO separate apps (runner + setup). There is no single `Munadi.app`. Any user-facing doc (Plan 06) that says "Munadi.app" as a generic term is OK as marketing copy, but the filesystem names are `MunadiRunner.app` + `MunadiSetup.app`.

<!-- Bundle identifier normalization. -->

Existing Info.plist files use `org.shoki.*` (not CONTEXT.md's `app.shoki.*` / `com.shoki.*`). We normalize on `org.munadi.*` to match the existing pattern. CONTEXT.md's `app.munadi.setup` / `com.munadi.runner` are superseded by:
- `org.munadi.setup` (Setup app, was `org.shoki.setup`)
- `org.munadi.runner` (Runner app + MachServices key, was `org.shoki.runner`)

Plan 01 (packages/sdk/src/cli/checks/tcc-grants.ts) expects `org.munadi.runner` / `org.munadi.setup`. This plan WRITES those Info.plists. Alignment is tight.

Ad-hoc signing (no Developer ID) continues through v0.1 per STATE.md Phase 10 Plan 03 decision. Do not change the signing identity in this plan.
